【译】 Apple Swift API 设计指南 Swift API Design Guidelines

Posted on 2019-04-24

Swift 语言 API 设计指南的中文翻译

基础原则

  • 使用时的清晰度是第一要义。评估一个设计时，仅仅读声明是不够的，要检查用例来确保它在上下文中看起来清晰。
  • 清晰度比简洁更重要。虽然 swift 语言可以写的很简洁，但是以最小字节数来写最短的代码永远不会是开发的目标。Swift 语言中的简洁性来自于强大的类型系统的副作用，和其他一些特性，这些特性能够减少样板代码。
  • 给每一个声明写文档注释。写文档注释会对设计产生巨大的影响，所以不要推迟。

如果你不能以简洁的语言描述 API 功能的话，说明 API 设计很可能有问题。

命名

促进清晰地使用

  • 包含所有需要的词来避免阅读代码的人产生歧义。
  • 去掉多余的词，每个词都要有突出的信息。
  • 给变量、参数和关联类型命名的时候要基于它们的角色，而不是基于它们的类型。
  • 补偿弱类型信息以阐明参数的作用。

力争流畅地使用

  • 使用方法和函数名，最好能让使用方形成有语法的英语惯用语。
  • 工厂方法使用 make 开头。
  • 初始化器和工厂方法的第一个参数不应该形成以基本名称开头的短语，e.g. x.makeWidget(cogCount: 47)
  • 基于函数和方法的副作用来命名：
    • 没有副作用的应该用名词，e.g. x.distance(to: y), i.successor().
    • 有副作用的应该用动词，e.g., print(x), x.sort(), x.append(y).
    • 始终如一地创建 mutating 和 non-mutating 方法对。一个 mutating 的方法常常有一个 non-mutating 的语义相近的变体。那个 non-mutating 变体返回一个新的值而不是直接更改当前实体
      • 如果一个运算是用动词来描述的，使用动词的 ed 或者 ing 后缀形式来命名它 non-mutating 的对应者。
      • 如果一个运算是用名词来描述的，那个名词给 non-mutating 方法命名。而使用 form 前缀来给 mutating 对应者来命名，e.g. y.union(z) => y.formUnion(z)
    • 如果是 non-mutating 的话，使用 Boolean 方法和属性应该读作断言一样，e.g. x.isEmpty, line1.intersects(line2).
    • 描述一个对象是什么的协议应该用名词
    • 描述一个对象能做什么的协议应该用 able, ible 或者 ing 后缀，e.g. Equatable, ProgressReporting.
    • 其他类型，属性，变量和常量的命名应该使用名词。

正确使用术语

Term of Art: 一个在特定领域或专业有着精确、特定含义的单词或词汇。

  • 如果一个常用词能表述的话，避免使用复杂的词语。例如，如果 skin 能够达到要求的话，不要使用 epidermis。Terms of Art 是必须的交流工具，但是应该用来抓住至关重要的含义，而不使用术语的话，那个含义可能会丢失。
  • 如果你使用术语的话，保持使用既定含义。
  • 避免缩写。缩写，特别是那些不标准的缩写，是一种有效的 terms-of-art，因为理解依赖于正确地解译。

你所使用的所有缩写的含义都应该很容易地在互联网上找到。

  • 拥抱先例。不要以牺牲与现有文化的一致性为代价来为初学者优化术语。

惯例

通用惯例

  • 给任何计算属性注释其运算复杂性。人们总是假定属性是没有明显的计算量的。
  • 尽量使用方法或者属性而不是自有函数。自有函数只在特殊情况下使用。
  • 类型和协议名使用大写开头的驼峰命名，其他的都使用小写开头的驼峰命名。
  • 方法可以使用相同的 base name，如果他们有相似的含义或者当它们操作在不同清晰的领域。

参数

  • 选择参数名来写文档，即使参数名不出现在使用方的方法或者函数上，它们对于解释非常有帮助
  • 使用默认参数，当它能够简化常用情况时。任何使用单一常用值的参数是默认值的候选。
  • 把默认参数放到参数列表的最后。没有默认值的参数通常对于方法的语义来说更重要，并且可以在方法被调用时，提供一个稳定的初始模式。

参数标签

  • 如果不能用于区分的话，移除标签，e.g. min(number1, number2), zip(sequence1, sequence2)
  • 在对值做类型转换的初始化器上，移除第一个标签。
  • 当第一个参数形成部分的介词短语的时候，使用参数标签。参数标签应该以介词开始，e.g. x.removeBoxes(havingLength: 12).
  • 否则，如果第一个参数形成语法短语的话，丢掉标签，但是给方法名后面加一个先行词。
  • 对其他任何情况的参数加上标签。

特别指南

  • 给元组成员和闭包参数加标签，当它们出现在你的 API 上的时候。
  • 小心无约束的多态类型，e.g. Any, AnyObject 和其他无约束的泛型参数。注意避免在重载的时候产生歧义。
